## Hallazgos críticos

### 🚨 CRÍTICO — SSRF en `/proxy-image` (PROXY ABIERTO)

**Archivo:** `ProxyController.java`

```java
@GetMapping
public ResponseEntity<byte[]> proxyImage(@RequestParam String url) {
    byte[] image = novaService.getNovaengelImage(url);
    // ...
}
```

El endpoint acepta **cualquier URL** arbitraria y la pasa directamente al cliente HTTP interno (`NovaApiClient` / `HttpClient`). Un atacante podría:

1. Hacer peticiones a servicios internos (`http://localhost:8080/admin/...`)
2. Escanear la red interna (`http://172.17.0.x:...`)
3. Acceder a metadatos de cloud (`http://169.254.169.254/`)

**No hay validación** de que la URL pertenezca a dominios разрешённые.

**Recomendación:** Validar que `url` sea de dominios permitidos (whitelist `drop.novaengel.com`, `images-na.ssl`, etc.) o usar un resolvedor DNS interno + validação de host.

---

### 🚨 CRÍTICO — Endpoint admin IMPORT sin autenticación JWT

**Archivo:** `ProductImportController.java`

```java
@PostMapping("/bts")
public ResponseEntity<String> importBts() {
    new Thread(btsService::importProductsBts).start();
    return ResponseEntity.ok("Importación de BTS iniciada...");
}
```

El controlador tiene `@CrossOrigin` pero **ningún `@AuthenticationPrincipal` ni validación de rol admin**. Cualquier usuario autenticado o incluso sin autenticar (si CORS permite) podría déclenchear importaciones masivas.

Los endpoints `POST /admin/import/bts`, `/admin/import/nova`, `/admin/import/nova/images`, `/admin/import/web/images` están bajo la regla `.requestMatchers("/admin/**").hasRole("ADMIN")` en `SecurityConfig`, lo que significa que **SÍ requieren ADMIN**. Esto está bien, pero conviene verificar que Spring security está correctamente configurado en este controller.

### ⚠️ ALTO — Posible bypass de CORS

**Archivo:** `SecurityConfig.java`

```java
configuration.setAllowedOrigins(List.of(
    "http://localhost:5173",
    "http://localhost:3000",
    // ... 11 origins including "http://localhost"
    "https://erosyafrodita.com",
    "https://erosyafrodita.com:8082"
));
```

Se permiten credenciales (`allowCredentials(true)`) con **11 orígenes permitidos incluyendo variantes de localhost**. Si una cookie de sesión o Authorization header se filtra a través de un subdomain.localhost, podría explotarse.

### ⚠️ ALTO — El JWT filter de tiendaBack hace una llamada Feign a mcsv-usuario en CADA request

**Archivo:** `JwtAuthFilter.java` (tiendaBack)

```java
UsuarioRegistroDto usuario = usuarioFeignClient.verUser(email);
```

Cada request autenticado genera una llamada HTTP síncrona al microservicio de usuarios para obtener el `UsuarioRegistroDto` completo. Esto:

1. Crea un punto único de fallo (si `mcsv-usuario` está lento, toda la API se ralentiza)
2. Añade latencia a cada request protegido
3. Satura la conexión entre servicios bajo carga

**Recomendación:** Cachear los datos del usuario en un TTL corto (Redis o en memoria) o incluir los datos necesarios (id, admin flag) directamente en el JWT.

## Hallazgos medios

### 🔸 MÉDIO — Feign client no tiene circuit breaker

**Archivo:** `UsuarioFeignClient.java`

El cliente Feign no tiene `@FeignClient(fallback = ...)` configurado. Si `mcsv-usuario` se cae, todas las requests protegidas a tiendaBack también fallan con error de microservicio en lugar de gracefully degrading.

### 🔸 MÉDIO — Posible enum bypass en PedidoEstado

**Archivos:** `PedidoServicieImpl.java`, `PedidoController.java`

```java
// En PedidoController
PedidoEstado estado = PedidoEstado.valueOf(body.get("estado"));
```

Si se envía un estado inválido, `valueOf()` lanza `IllegalArgumentException` que es capturada y retorna 400. Esto está bien, pero el enum `PedidoEstado` podría no cubrir todos los estados posibles del sistema proveedor.

### 🔸 MÉDIO — No hay rate limiting en endpoints de carrito o reseñas

Los endpoints `POST /carrito/agregar`, `GET /resenas/producto/{id}` no tienen protección contra滥用 (abuse). Un usuario podría spammear requests de carrito o listar reseñas masivamente.

### 🔸 MÉDIO — `@CrossOrigin` duplicado en varios controllers

`ProductoController`, `ProductImportController` y `ProxyController` tienen `@CrossOrigin(origins = ...)` hardcoded. Esto es difícil de mantener — considerar centralizar la config CORS en un filtro único.

## Hallazgos menores

### • MINOR — Tokens JWT sin rotate explícito

Los tokens JWT actuales expiran en 1 hora y se asumen vigentes hasta entonces. No hay refresh token implementado. Si un token se compromete, permanece válido hasta su expiración natural. La blacklist implementada ayuda, pero no existe un mecanismo de refresh.

### • MINOR — Password encoder BCrypt strength 12 en mcsv-usuario (OK) pero no hay verificación en tiendaBack

Solo `mcsv-usuario` tiene `BCryptPasswordEncoder` bean. `tiendaBack` también declara un `BCryptPasswordEncoder` bean pero no está claro si se usa para algo (no hay registro de usuarios en tiendaBack).

### • MINOR — Logs sensibles en RevolutWebhookController

```java
log.info("Recibido Webhook de Revolut: Evento={}, Status={}", ...);
```

Se loguea el payload completo del webhook de pago. Si el logs se almacenan en texto plano, podrían contener datos financieros sensibles.

### • MINOR — Posible DOS por `new Thread()` en ProductImportController

Cada llamada a `/admin/import/bts` crear un nuevo Thread sin límite de concurrencia. Si se llama múltiples veces, podría agotar el thread pool de la JVM.

### Fix Login "Not Found" (404)
**Causa:** La regla nginx `location /api/auth/login` con `proxy_pass http://tienda-app-server:8080/` reenviaba mal el path al backend Spring.
- Backend tiene endpoints en `/auth/login` (no `/api/auth/login`)
- El trailing slash causaba que el 404 viniera del backend (ruta no existe)

**Solución:**
```nginx
location /api/auth/login {
    rewrite ^/api(/.*)$ $1 break;   # /api/auth/login → /auth/login
    proxy_pass http://tienda-app-server:8080;  # sin trailing slash
}
location /api/ {
    rewrite ^/api(/.*)$ $1 break;   # cobertura general
    proxy_pass http://tienda-app-server:8080;
}
```
